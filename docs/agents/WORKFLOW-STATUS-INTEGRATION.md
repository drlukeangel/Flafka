# Workflow Status: Template + Live Data Integration

**Complete guide to how workflow status is structured, generated, and maintained**

---

## System Overview

Three interconnected parts work together to provide real-time workflow visibility within self-contained feature run folders:

```
docs/agents/
├─ workflow-status-template.md (STATIC)
│  └─ Schema & structure definition (source for all runs)
│
└─ feedback/
   └─ run-{N}/  ← Each feature run is self-contained
      ├─ workflow-status.md (LIVE)
      │  └─ Populated every 60 seconds by Workflow Manager
      │  └─ Single source of truth for current run state
      │
      └─ [PERMANENT AUDIT TRAIL]
         ├─ TPPM.md
         ├─ QA-MANAGER.md
         ├─ PRINCIPAL-ARCHITECT.md
         ├─ ENGINEERING.md
         ├─ ... (all agent feedback files)
         └─ Never deleted — permanent audit trail
```

---

## File Purposes

### 1. workflow-status-template.md (Schema Definition)

**What it is:** Static template defining the structure and sections of the live status file

**Who maintains it:** Workflow Manager reads this to understand required structure; updated only when schema changes

**Key sections:**
- Header (timestamp, cycle number)
- Active Features (table structure)
- Running Agents (table structure)
- Gate Status (table structure)
- A2 Design Review Tracker
- Non-Blocking Engineering Notes
- UX/IA Phase 2.6 Conditions
- A3 PRD Revision Tracker
- Violations & Alerts
- Feature Organizer & Ranker Alignment
- Recently Completed Features
- Feedback Pipeline
- Gate Enforcement Summary
- Next Recommended Actions

**Rules:**
- ✅ Read by Workflow Manager every cycle
- ✅ Updated when gate/section schema changes
- ❌ Never contains live data (only empty table templates)
- ❌ Not edited manually (auto-generated from agent behavior)

---

### 2. workflow-status.md (Live Data per Run)

**What it is:** Live data file for a specific feature run, populated every 60 seconds by Workflow Manager

**Location:** `docs/agents/feedback/run-{N}/workflow-status.md` (same folder as agent feedback files)

**Who maintains it:** Workflow Manager (automated, continuous)

**Contents:** Copy of template structure, populated with current run data:
- Actual feature name, phases, statuses for this run
- Real agent heartbeats and ETAs
- Current gate status (cleared/pending/blocked)
- Active violations and blockers
- Concrete next action items
- Links to agent feedback files in same folder

**Rules:**
- ✅ Updated automatically every 60 seconds
- ✅ Updated immediately on phase gate transitions
- ✅ Single source of truth for current run state
- ✅ Claude Code reads this before major decisions
- ✅ Lives in run-{N}/ folder alongside agent feedback files
- ❌ Never edited manually (auto-generated by Workflow Manager)
- ❌ File is completely overwritten each cycle (full refresh)

---

### 3. docs/agents/feedback/run-{N}/ (Self-Contained Feature Run Folder)

**What it is:** Self-contained folder for a feature run containing both live status + permanent audit trail

**Who maintains it:**
- Workflow Manager creates folder, copies template, populates workflow-status.md
- Each agent writes feedback as checkpoints complete (asynchronously)

**Structure:**
```
docs/agents/feedback/
├── run-1/        ← Feature 1 cycle (self-contained)
│   ├── workflow-status.md  ← LIVE DATA (auto-populated by Workflow Manager)
│   ├── TPPM.md             ← Agent feedback
│   ├── QA-MANAGER.md
│   ├── PRINCIPAL-ARCHITECT.md
│   ├── UX-IA-REVIEWER.md
│   ├── PRINCIPAL-ENGINEER.md
│   ├── ENGINEERING.md
│   ├── CLOSER.md
│   ├── VALIDATION-TESTING.md
│   ├── TEST-COMPLETION.md
│   ├── FEATURE-ORGANIZER-RANKER.md
│   ├── WORKFLOW-MANAGER.md
│   └── AGENT-DEFINITION-OPTIMIZER.md
├── run-2/        ← Feature 2 cycle (self-contained)
│   ├── workflow-status.md  ← LIVE DATA (auto-populated by Workflow Manager)
│   ├── TPPM.md
│   └── [same agent feedback files]
├── run-3/        ← Feature 3 (etc.)
└── README.md     ← Purpose & format guide (shared across all runs)
```

**Feedback format per agent:**
```markdown
# [AgentName] Feedback

**Reference Profile:** [docs/agents/AGENT-NAME.md](../../AGENT-NAME.md)
**Feature Cycle:** Run {N}
**Timestamp:** [ISO 8601]

## Definition vs Reality
**What my MD says I do:** [quote]
**What I actually did:** [description]
**Aligned?** Yes / No / Partially

## Gaps Found
- Missing from definition: [capability]
- Outdated assumption: [change]
- Suggestion: [how MD should be updated]

**Confidence:** High/Medium/Low
```

**Rules:**
- ✅ Agents write asynchronously when milestones complete
- ✅ Each run gets incrementing folder (run-1, run-2, run-3...)
- ✅ All agents contribute feedback (even if "no changes needed")
- ✅ NEVER DELETED — permanent audit trail
- ✅ Workflow Manager reads latest feedback every 60s
- ✅ Enables Agent Definition Optimizer to detect patterns
- ❌ Not edited after written (immutable records)
- ❌ Do NOT commit directly to git; let Closer handle batch commits

## Data Flow

### Workflow Manager 60-Second Cycle

```
START (every 60 seconds)
  ↓
1. Determine current run number (N) from roadmap.md
  ↓
2. Check if run-{N} folder exists:
   - If NO: Create docs/agents/feedback/run-{N}/ folder
   - If NO: Copy workflow-status-template.md → docs/agents/feedback/run-{N}/workflow-status.md
   - If YES: Continue to next step
  ↓
3. Scan docs/agents/feedback/run-{N}/ for latest agent reports
  ↓
4. Parse each agent's [AgentName].md:
   - Extract status (active/blocked/complete/idle)
   - Extract task description
   - Extract phase assignment
   - Extract blockers and notes
   - Extract ETA
   - Extract timestamp (for staleness check)
  ↓
5. Read docs/agents/workflow-status-template.md (structure schema)
  ↓
6. Query roadmap.md for active features + phase status
  ↓
7. Validate gate conditions from:
   - A2 Design Review Tracker (all 5 reviewers)
   - Phase gate validation logs
   - Feature acceptance status
  ↓
8. Identify violations:
   - Stale agent feedback (> 120 seconds old)
   - Phase progression without gate approval
   - Blocker escalations
   - Missing feedback from expected agents
  ↓
9. Consolidate recommendations from all agent feedback
  ↓
10. Populate template structure with live data:
    - Header: current timestamp + cycle # + run folder name
    - Active Features: from roadmap + feedback
    - Running Agents: from feedback folder (latest reports)
    - Gate Status: from gate validation + feedback
    - Violations & Alerts: from logs + feedback
    - Recommendations: consolidated from all agents
  ↓
11. Write complete file to docs/agents/feedback/run-{N}/workflow-status.md
    (OVERWRITE — full refresh each cycle)
  ↓
12. Log write timestamp + section summary
  ↓
WAIT 60 SECONDS, REPEAT
```

## Integration Example (Current State)

**Scenario:** Ops Excellence Drill-Down is in Phase B1 (implementation), Run-7 is active

**Data flow:**

1. **Agent writes feedback** (async):
   - Engineering writes: `docs/agents/feedback/run-7/ENGINEERING.md`
   - Contains: "B1 Implementation ACTIVE, 85% complete, 3 non-blocking notes being addressed"
   - Timestamp: 2026-02-28T21:00:00Z

2. **Workflow Manager polls** (every 60 seconds):
   - Determines current run: run-7 (from roadmap.md)
   - Reads: `docs/agents/feedback/run-7/ENGINEERING.md`
   - Parses: status=ACTIVE, task="B1 Implementation", eta="2026-03-08", timestamp=21:00:00Z
   - Checks: is timestamp recent (< 120s old)? YES ✅

3. **Workflow Manager reads template**:
   - Template says "Running Agents" section needs columns: Agent, Status, Task, Phase, Last Heartbeat, ETA

4. **Workflow Manager queries roadmap.md**:
   - Finds: Ops Excellence Drill-Down in Phase 2 — B1 active

5. **Workflow Manager generates live data**:
   - Populates: `docs/agents/feedback/run-7/workflow-status.md` (SAME FOLDER as feedback files)
   - Running Agents section shows:
     ```
     | Agent | Status | Task | Phase | Last Heartbeat | ETA |
     | Engineering | ✅ ACTIVE — B1 Implementation | A2 gate cleared. B1 implementation in progress. | Phase 2 — B1 | 2026-02-28T21:00:00Z | 2026-03-08 |
     ```

6. **Claude Code reads live file**:
   - Before deciding what to do next, reads: `docs/agents/feedback/run-7/workflow-status.md` (active run folder)
   - Sees: Engineering is actively implementing, B1 should complete by 2026-03-08
   - Proceeds with appropriate next action

## Staleness & Alert Detection

**Workflow Manager monitors agent responsiveness:**

| Scenario | Action |
|----------|--------|
| Agent feedback timestamp < 120 seconds old | ✅ RESPONSIVE — show in Running Agents, status current |
| Agent feedback timestamp 120-300 seconds old | ⚠️ STALE — show in Running Agents, mark as "stale heartbeat" |
| Agent feedback timestamp > 300 seconds old or missing | ❌ UNRESPONSIVE — flag in Violations section, alert Claude Code |
| New agent feedback written while scanning folder | ✅ Detected immediately on next cycle (60s latency, not an issue) |

## Rules & Guarantees

### Workflow Manager Responsibilities
- ✅ Create run-{N}/ folder at start of each feature (if doesn't exist)
- ✅ Copy workflow-status-template.md → run-{N}/workflow-status.md (on first cycle of run)
- ✅ Update `run-{N}/workflow-status.md` every 60 seconds without fail
- ✅ Update immediately on any phase gate transition
- ✅ Parse feedback folder (run-{N}/) correctly and extract agent status
- ✅ Never exit loop (run continuously forever)
- ✅ Handle errors gracefully (log error, wait 60s, retry)
- ✅ Keep running even if no features are active (idle state)

### Agent Responsibilities
- ✅ Write feedback to `run-{N}/[AgentName].md` asynchronously as milestones complete
- ✅ Include timestamp, status, task description, blockers, next steps
- ✅ Write honestly (definition vs reality, confidence level)
- ✅ Don't wait for Workflow Manager to ask (write when ready)
- ✅ Feedback files are immutable (never edit after written)

### Closer Responsibilities
- ✅ Commit feedback files in batch at Phase 4A
- ✅ NEVER DELETE `docs/agents/feedback/` folder
- ✅ Organize runs into meaningful feature cycles
- ✅ Update `run-N/` numbering for each new feature

### Claude Code Responsibilities
- ✅ Read `docs/agents/feedback/run-{N}/workflow-status.md` (active run) before major decisions
- ✅ Check gate status before phase transitions
- ✅ Check blockers before scheduling work
- ✅ Use "Next Recommended Actions" section to guide priorities
- ✅ Never bypass workflow status (always consult live file in active run folder)
